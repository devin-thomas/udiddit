From a fragile dump to a reviewer-ready relational design.

This Udiddit project rebuilds a denormalized social news schema into a clean PostgreSQL data model, migrates the supplied source data, and packages the work in a submission-ready report and portfolio site.

The goal was not just to satisfy the rubric, but to make the reasoning behind the schema, migration order, and validation story quick to understand on the web.

Schema Design

The redesign addresses the three biggest weaknesses in the starter schema: duplicated text entities, comma-separated votes, and flat comments. The new model adds real constraints, intentional foreign-key behavior, and indexes shaped around the moderation and activity queries from the rubric.

What the redesign fixes

  • Entity duplication Usernames and topic names move out of free-text columns and into dedicated tables with uniqueness enforced at the database level.
  • Votes in first normal form Each vote becomes its own row, making one-vote-per-user enforceable and making score queries straightforward and index-friendly.
  • Thread-ready comments Comments gain a self-reference so the platform can support arbitrary nesting instead of only a flat reply list.

Schema at a glance

Deletes cascade where content should disappear together, and user references fall back to NULL where the rubric calls for dissociation instead of removal.

users Unique usernames, optional last-login tracking
topics Unique topic names and optional descriptions
posts Exactly one of URL or text content, linked to topic and user
comments Self-referential threads scoped to a post
votes One row per user-post vote with values of 1 or -1
Named constraints + targeted indexes

Migration Strategy

The migration script follows the dependency chain of the new model: users first, then topics, posts, comments, and votes. That sequencing keeps every foreign-key reference valid and preserves the original post and comment IDs from the starter dump.

Why the order matters

  • Users first The migration collects authors, commenters, and vote participants so no downstream insert depends on a missing account.
  • Topics before posts Posts resolve topic IDs through the normalized topics table rather than duplicating topic names on every row.
  • Votes last Votes are exploded from the old comma-separated text fields only after both posts and users are in place.

Data-quality tradeoff documented explicitly

The supplied dataset contains 152 legacy titles longer than the stated 100-character product rule. Instead of truncating source data, the final schema stores titles up to 150 characters and documents that mismatch in the submission report.

That keeps the migration lossless while still making the discrepancy visible to a reviewer.

Validation Story

The finished project includes a dedicated validation script and example DQL set so the reviewer can confirm counts, integrity rules, and rubric-driven query behavior without digging through the migration by hand.

What was checked

  • Source-to-target counts Users, topics, posts, comments, and votes all match the expected totals after migration.
  • Post content rule Every migrated post contains either a URL or text content, but never both and never neither.
  • Vote integrity Votes are restricted to 1 and -1, and no non-null user can vote twice on the same post.
  • Stand-out query coverage The repo includes a JSON comment-tree query with three levels of nesting in addition to the rubric’s core DQL examples.

Final verified totals

  • 11,077 users Unified from post authors, comment authors, and vote participants.
  • 89 topics Normalized from repeated topic-name text values.
  • 50,000 posts / 100,000 comments Starter content preserved with original post and comment identifiers.
  • 499,710 votes Exploded safely from the denormalized upvote and downvote text columns.
